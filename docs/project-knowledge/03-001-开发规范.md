## 三、代码封装规范

### 规范 1：分层职责清晰——Controller 仅做编排，Service 持业务逻辑，Mapper 仅继承 BaseMapper 不写 XML

- **内容**：
  - **Mapper**：接口加 `@Mapper` 注解、继承 MyBatis-Plus `BaseMapper<Entity>`，不写 Mapper XML（全项目无任何 `*Mapper.xml` 文件），复杂查询统一用 `LambdaQueryWrapper` 在 Service 层构造。
  - **Service**：接口继承/声明业务方法，实现类继承 `ServiceImpl<XxxMapper, XxxEntity> implements XxxService`，业务异常通过 `throw new BusinessException(ErrorCode.XXX)` 抛出。
  - **Controller**：`@RestController` + `@RequestMapping` + `@RequiredArgsConstructor`，仅做参数接收、调用 Service、组装 `ApiResponse` 返回，跨业务编排可注入多个 Service。
- **证据**：
  - Mapper：`mapper/SysUserMapper.java` —— `@Mapper public interface SysUserMapper extends BaseMapper<SysUser> {}`；全量 `find dsp-core/src/main/resources/mapper -name "*.xml"` 返回空（无 XML）
  - ServiceImpl：15 个实现类全部 `grep -c "extends ServiceImpl"` = 1，如 `InterfaceInfoServiceImpl extends ServiceImpl<InterfaceInfoMapper, InterfaceInfo> implements InterfaceInfoService`
  - 业务异常抛出：`service/impl/InterfaceInfoServiceImpl.java:42` —— `if (info == null) throw new BusinessException(ErrorCode.INTERFACE_NOT_FOUND);`、`:51` —— `throw new BusinessException(ErrorCode.INTERFACE_NOT_FOUND, "接口模板配置不存在");`
  - Controller 编排：`controller/InterfaceAdminController.java` —— 注入 `InterfaceInfoService interfaceInfoService`、`InterfaceVersionService interfaceVersionService`、`ApprovalRecordService approvalRecordService`、`XmlEngine xmlEngine` 等多个 Service 协作
  - LambdaQueryWrapper 用法：`InterfaceInfoServiceImpl.java:24-28` —— `LambdaQueryWrapper<InterfaceInfo> wrapper = new LambdaQueryWrapper<>(); wrapper.eq(InterfaceInfo::getTransno, transno).eq(InterfaceInfo::getStatus, ...); return getOne(wrapper);`
- **置信度**：高（15 个 ServiceImpl + 18 个 Mapper + 17 个 Controller 模式一致）
- **来源类型**：代码推断

### 规范 2：依赖注入首选构造器注入（`@RequiredArgsConstructor`），仅循环依赖用 `@Lazy @Autowired` 字段注入

## 五、错误处理

### 规范 1：统一业务异常 `BusinessException` + 错误码枚举 `ErrorCode`，业务层抛异常不返回错误码

- **内容**：业务逻辑遇到异常情况统一 `throw new BusinessException(ErrorCode.XXX)` 或 `throw new BusinessException(ErrorCode.XXX, "补充信息")`，不在业务方法里手工构造错误响应；`BusinessException` 携带 `code` + `message`，构造器接收 `ErrorCode` 枚举或裸字符串 code。
- **证据**：
  - `dsp-common/.../exception/BusinessException.java`：`@Getter public class BusinessException extends RuntimeException { private final String code; public BusinessException(ErrorCode errorCode) {...} public BusinessException(ErrorCode errorCode, String message) {...} public BusinessException(String code, String message) {...} }`
  - 业务抛出：`InterfaceInfoServiceImpl.java:42` `throw new BusinessException(ErrorCode.INTERFACE_NOT_FOUND);`、`:51` `throw new BusinessException(ErrorCode.INTERFACE_NOT_FOUND, "接口模板配置不存在");`
  - 角色校验抛出：`aspect/RoleCheckAspect.java:33,49` `throw new BusinessException(ErrorCode.ACCESS_DENIED, "无操作权限");`
- **置信度**：高（异常类定义 + 多处 Service/Aspect 一致抛出）
- **来源类型**：代码推断

### 规范 2：每个可部署服务独立 `@RestControllerAdvice` 全局异常处理器，统一映射 HTTP 状态码

- **内容**：三个服务各自在 `handler/GlobalExceptionHandler.java` 用 `@RestControllerAdvice` + `@ExceptionHandler` 收口异常：`BusinessException` 按 code 映射 HTTP 状态（TOKEN_MISSING/TOKEN_EXPIRED→401、ACCESS_DENIED→403、BAD_REQUEST→400、其余业务异常→200 用 code 区分）、参数校验异常（`MethodArgumentNotValidException`/`BindException`/`MissingServletRequestParameterException`/`MethodArgumentTypeMismatchException`/`HttpMessageNotReadableException`）统一 400、兜底 `Exception` 500；data-service 另有 `GlobalResponseBodyAdvice` 做统一响应体包装。
- **证据**：
  - `dsp-admin-service/.../handler/GlobalExceptionHandler.java:23-44`：`@ExceptionHandler(BusinessException.class)` 内分支判断 `ErrorCode.TOKEN_MISSING/TOKEN_EXPIRED → 401`、`ACCESS_DENIED → 403`、`BAD_REQUEST → 400`、其余 `ResponseEntity.ok(...)`
  - 同文件 `:46-72`：`@ExceptionHandler({MethodArgumentNotValidException.class, BindException.class, ...}) @ResponseStatus(HttpStatus.BAD_REQUEST)`
  - 同文件 `:74-79`：`@ExceptionHandler(Exception.class) @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR) ... ApiResponse.error(..., ErrorCode.SYSTEM_ERROR)`
  - `dsp-data-service/.../handler/GlobalExceptionHandler.java`、`dsp-offline-service/.../handler/GlobalExceptionHandler.java` 同构存在
  - `dsp-data-service/.../handler/GlobalResponseBodyAdvice.java` 做响应体统一包装
- **置信度**：高（3 个服务同构 + HTTP 状态映射规则在代码中明文）
- **来源类型**：代码推断

### 规范 3：日志统一 Lombok `@Slf4j`，按场景分级（warn=业务/参数异常、error=系统异常、info=正常流程、debug=调试）

- **内容**：类级 `@Slf4j` 注解引入日志器（全后端 56 个类使用）；日志分级约定：业务异常/参数异常用 `log.warn`（不打满栈）、系统兜底异常用 `log.error`（带异常对象）、正常关键流程用 `log.info`、调试细节用 `log.debug`。
- **证据**：
  - `@Slf4j` 分布：56 个 .java 类标注
  - 分级统计（grep 全后端 main 源码）：`log.warn` 51 处、`log.info` 50 处、`log.debug` 23 处、`log.error` 18 处
  - 样例：`GlobalExceptionHandler.java:25` `log.warn("业务异常: code={}, message={}", ...)`、`:70` `log.warn("参数异常: type={}, message={}", ...)`、`:77` `log.error("系统异常", e)`（带异常对象）；`RoleCheckAspect.java:48` `log.warn("权限不足: userRoles={}, required={}", ...)`
- **置信度**：高（56 类 + 142 处日志调用，分级规律清晰）
- **来源类型**：代码推断

## 七、Git 提交规范

### 规范 1：Commit Message 格式 `<type>: <中文描述>`，9 种 type 清单

- **内容**：提交信息遵循 `<type>: <中文描述>` 格式；type 取自固定清单：`feat`（新功能）、`fix`（Bug 修复）、`docs`（文档）、`style`（格式）、`refactor`（重构）、`perf`（性能）、`test`（测试）、`chore`（构建/工具/配置）、`build`（CI/CD）；描述用中文，PR 标题与 commit message 格式一致。
- **证据**：
  - `CONTRIBUTING.md` 第 2 节显式声明 Commit Message 格式 `<type>: <中文描述>` + 9 种 type 表格 + 中文示例（`feat: 新增审批管理页面`、`fix: 修复分页参数解析异常`）
  - git log 实际提交（40 条）type 分布：`feat` 19、`fix` 9、`refactor` 5、`docs` 3、`style` 2，均为 `<type>: <中文描述>` 格式
  - 样例：`733a0e6 feat: 项目改进路线图 T01-T18 全部实现`、`f5289c8 docs: 修正引擎文档与代码不一致的内容 (v2)`、`46e3e81 fix: 接口申请提交报错 version_no 没有默认值 (#47) (#48)`、`a329ebe style: 前端UI布局优化 — 顶部栏+侧边栏调整 (#47)`
- **置信度**：高（显式文档声明 + 38/40 条历史提交严格遵守，仅 2 条异常：「111 (#53)」与一条 Revert 反引号包裹）
- **来源类型**：混合（CONTRIBUTING.md 显式 + git 历史一致）

### 规范 2：分支命名 `feature/<type>-<desc>-<date>` / `hotfix/<desc>-<date>` / `release/<version>`

- **内容**：开发分支用 `feature/<type>-<desc>-<date>`（`type` = feat/fix/refactor/docs/perf），紧急修复 `hotfix/<desc>-<date>`（合并后必须打 tag），发布候选 `release/<version>`（冻结功能仅修 bug）；`main` 为生产分支，禁止 push，仅接受 PR，使用 Squash and Merge。
- **证据**：
  - `CONTRIBUTING.md` 第 1 节显式声明分支策略表 + 示例 `feature-feat-approval-flow-20260430`、`hotfix-jwt-expiry-20260501`、`release/v1.1.0`
  - git 分支实际样例：`feature-refactor-migrate-codex-20260526`、`feature-feat-approval-refactor-20260518`、`feature-feat-history-exclude-current-20260514`、`fix-approval-version-no-default-20260516`、`fix-pending-approval-sql-20260516`、`feature/cicd-pipeline-20260510`、`refactor-package-rename-20260512`、`codex-improvement-roadmap`
  - 注意：实际分支存在两种连字符风格——`feature-feat-desc-date`（CONTRIBUTING 规范的 `/` 用 `-` 替代后整体）与 `feature/desc-date`（用 `/`），部分早期分支（`feature-sunlc-20260429`、`fix-approval-version-no-default-20260516`）不完全带 type 前缀，存在少数例外
- **置信度**：中（显式文档声明为高，但实际分支命名存在 `-` vs `/` 与是否带 type 的不一致，标注「多数一致，存在例外」）
- **来源类型**：混合

### 规范 3：版本号 Semantic Versioning `vMAJOR.MINOR.PATCH`

- **内容**：版本号遵循 SemVer，MAJOR=不兼容 API 变更（如 XML DSL 语法升级）、MINOR=向后兼容新功能、PATCH=向后兼容 Bug 修复；发布流程：创建 `release/vX.Y.Z` 分支 → 冻结功能 → 修复问题 → 打 tag → 合并回 main → 生成 CHANGELOG。
- **证据**：`CONTRIBUTING.md` 第 4 节显式声明「采用 Semantic Versioning：`vMAJOR.MINOR.PATCH`」+ MAJOR/MINOR/PATCH 语义定义 + 发布流程
- **置信度**：高（显式文档声明；jar 产物命名 `dsp-*-1.0.0.jar` 与 1.0.0 版本一致）
- **来源类型**：显式文档

### 规范 4：PR 控制单次变更 ≤400 行，至少 1 人 review，需编译+测试通过

- **内容**：单个 PR 变更控制在 400 行以内（大型功能拆分多 PR）；PR 前置条件包括描述填写完整（变更类型 checklist + 影响范围 + 测试验证 + 风险说明）、至少 1 人 review 批准、无未解决评论、编译通过 + 测试通过；审查清单覆盖功能正确性/代码规范/安全性/性能/测试覆盖/兼容性。
- **证据**：`CONTRIBUTING.md` 第 3 节显式声明「单个 PR 变更控制在 400 行以内」「至少 1 人 review 批准」「编译通过 + 测试通过」+ 审查清单 6 项
- **置信度**：高（显式文档声明）
- **来源类型**：显式文档
