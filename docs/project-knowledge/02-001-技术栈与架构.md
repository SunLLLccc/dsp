## 二、项目自封装框架

> 以下为项目自建的内部框架/SDK/基类/中间件，与开源第三方库明确区分。**核心自封装是 `dsp-engine` 模块下的 XML DSL 查询引擎**，这是整个项目"零代码接口发布"能力的载体。

### 2.1 XmlEngine — XML DSL 查询编排引擎（核心自封装）

- **所在模块/目录**：`dsp-parent/dsp-engine/src/main/java/com/sunlc/dsp/engine/`
- **设计意图**：将"接口逻辑"从 Java 代码剥离到 XML 配置（DSL）中，业务方写 XML 即可发布一个查询/导出接口，无需改后端代码；引擎负责解析 XML → 参数校验 → DAG 并行执行多查询 → 结果映射 → 响应构建。
- **核心组件**：

| 组件 | 文件路径（相对 `dsp-engine/src/main/java/com/sunlc/dsp/engine/`） | 职责 |
|------|---------|------|
| `XmlEngine` | `XmlEngine.java` | 引擎门面，串联"参数校验→注册内联数据源→查询编排→结果映射→响应构建"四阶段；带 `DebugContext` 零开销调试跟踪 |
| `XmlConfigParser` | `parser/XmlConfigParser.java` | DOM4J 解析 XML → `InterfaceConfig` 对象树（requestData/datasource/query/resultMap/responseData） |
| `QueryOrchestrator` | `executor/QueryOrchestrator.java` | **DAG 编排器**：基于 `depends` 依赖声明，用 `CompletableFuture` + 线程池（core=4/max=8）并行执行查询；含依赖校验 + 环检测（DFS 三色标记） |
| `SqlExecutor` | `executor/SqlExecutor.java` | SQL 查询执行器：用 `JdbcTemplate` + `DynamicDataSourceContextHolder.push(datasource)` 切库 |
| `HttpExecutor` | `executor/HttpExecutor.java` | `http` 类型查询：按配置发 HTTP 请求并归一化结果 |
| `DubboExecutor` | `executor/DubboExecutor.java` | `dubbo` 类型查询：经 ZK/Curator 发现 provider 并 RPC 调用 |
| `MongoExecutor` | `executor/MongoExecutor.java` | `mongo` 类型查询（可选 Bean，未引 mongodb starter 时为 null） |
| `DynamicSqlHandler` | `executor/DynamicSqlHandler.java` | **动态 SQL 处理器**：解析 `<if>`/`<foreach>`，用 Spring SpEL 评估条件，处理 `#{param}` 占位符 → 参数化 SQL（防注入） |
| `PaginationHandler` | `executor/PaginationHandler.java` | 分页改写器：支持 `OPTIMIZED`（页码）/`CURSOR`（游标）两种模式改写 SQL，含导出批次模式 |
| `ResultMapper` | `executor/ResultMapper.java` | 结果映射：`resultMap` 字段重命名/函数加工 + `responseData` 响应结构组装 |
| `FunctionRegistry` | `function/FunctionRegistry.java` | DSL 内置函数注册表（`DATE_FORMAT`/`CONCAT`/`TYPE_CONVERT` 等），供 `ResultMapper` 调用 |
| `SqlSecurityValidator` | `validator/SqlSecurityValidator.java` | 接口发布前校验 SQL 只读（拦截 UPDATE/DELETE/DROP 等 12 个关键字 + 注释 + 分号 + orderBy 白名单） |
| `XmlConfigCacheManager` | `cache/XmlConfigCacheManager.java` | `transno → InterfaceConfig` 内存缓存（ConcurrentHashMap），未命中从 DB 加载解析；实现 `XmlConfigCacheInvalidator` 接口 |
| `DataSourceRegistrar` | `DataSourceRegistrar.java` | 回调接口：XML 内联 `<datasource>` 由各服务（如 `InlineDataSourceRegistrar`）注入实现，动态注册数据源 |
| `DebugContext` / `DebugTrace` | `DebugContext.java`、`model/DebugTrace.java` | 调试上下文与单查询 trace（SQL/参数/行数/耗时），驱动"接口调试"页面 |
| `model/*Config.java` | `model/`（17 个配置类） | XML 元素的对象化：`InterfaceConfig`、`QueryConfig`、`PaginationConfig`、`DataSourceConfig`、`ResultMapConfig` 等 |

- **被依赖方**：
  - `dsp-data-service` 的 `DataApiController`（对外查询端点 `POST /dsp/api/{transno}`）直接调用 `XmlEngine.executeWithConfig`
  - `dsp-offline-service` 通过 `DataQueryService` 间接复用引擎做导出
  - `dsp-admin-service` 的接口调试/发布校验路径调用 `SqlSecurityValidator` 与引擎执行
- **框架级特征印证**：独立模块、内部包结构（`executor/parser/validator/function/cache/model/service`）、被 3 个服务模块依赖、提供统一执行门面、含自定义模型对象树与缓存管理、依赖 Spring 容器装配（`@Component`）。

### 2.4 后端横切机制（鉴权 / 审计 / 角色控制）— 自封装注解 + AOP

这套机制散落在各服务包下，但属于项目自封装的统一横切设计，故合并记录。

- **设计意图**：用 Spring AOP + 自定义注解/切面实现对外 API 的 JWT 鉴权、后台 RBAC、操作审计，避免侵入业务代码。
- **核心组件**：

| 组件 | 文件路径 | 职责 |
|------|---------|------|
| `JwtAuthAspect` | `dsp-data-service/.../interceptor/JwtAuthAspect.java` | 对外 API 切面：校验 JWT、`appId↔token` 一致性、`allowedTransnos` 接口白名单、`timestamp` ±5 分钟防重放；`token="base"` 跳过（调试用） |
| `JwtAuthInterceptor` | `dsp-offline-service/.../interceptor/JwtAuthInterceptor.java` | 离线导出服务相同语义的 JWT 拦截器（HandlerInterceptor 形态） |
| `AdminAuthInterceptor` + `WebMvcConfig` | `dsp-admin-service/.../interceptor/AdminAuthInterceptor.java` | 后台登录态拦截：校验 `Admin-Token` header，解析后将 `adminUser/adminUserId/adminRoles/adminDeptId` 注入 request attribute |
| `@RequireRole` + `RoleCheckAspect` | `dsp-admin-service/.../annotation/RequireRole.java`、`aspect/RoleCheckAspect.java` | 方法级 RBAC：`@RequireRole({"USER","DEPT_MANAGER"})`，ADMIN 角色全放行，否则需命中其一；前端配套 `directives/role.js` 指令做按钮级控制 |
| `AuditLogAspect` | 各服务的 `interceptor/AuditLogAspect.java` | 操作审计日志切面（写 `audit_log` 表） |

- **被依赖方**：各服务自身的 Controller 与 admin 前端（`Admin-Token`、`v-role` 指令）。
- **框架级特征印证**：自定义注解、统一 AOP 切面、跨前后端协同（前端指令与后端注解同名语义）、被多个 Controller 引用。

### 分层结构

后端每个可部署服务采用经典的四层 + 横切切面结构：

| 层 | 职责 | 代表组件 |
|----|------|----------|
| **Controller 层** | 接收请求、报文解包、调用 Service/Engine、统一返回 `ApiResponse` | `DataApiController`、`InterfaceAdminController` 等 |
| **AOP/Interceptor 横切层** | 鉴权（对外 JWT / 后台 Admin-Token）、RBAC（`@RequireRole`）、审计日志、全局异常/响应体增强 | `JwtAuthAspect`、`RoleCheckAspect`、`GlobalExceptionHandler`、`GlobalResponseBodyAdvice` |
| **Service 层** | 业务编排；管理域走 MyBatis-Plus `XxxServiceImpl`，查询域走 `XmlEngine` | `InterfaceInfoServiceImpl`、`DatasourceManagerService`、`ExportServiceImpl`、`XmlEngine` |
| **持久/执行层** | MyBatis-Plus Mapper（管理域）+ `JdbcTemplate` 动态数据源（DSL SQL 执行域）+ HTTP/Dubbo/Mongo Executor | `XxxMapper`、`SqlExecutor`、`DubboExecutor` |
| **数据层** | MySQL `dsp_config`（元数据）+ 业务库（动态数据源）+ Redis（缓存）+ MongoDB（可选查询源） | `application.yml` |

前端分层：`views`（页面）→ `components`（可复用组件）→ `api`（HTTP 封装）→ `stores`（Pinia 状态）→ `router`（路由+守卫）→ `directives`（`v-role`）。

### 数据流

**核心查询数据流（对外 API `POST /dsp/api/{transno}`）：**

1. 客户端发送 `ApiRequest`（`head.token/appId/timestamp/traceId` + `body.requestData`）。
2. `JwtAuthAspect`（切 `DataApiController.*`）前置校验：JWT 解析 → `appId↔token` 比对 → `allowedTransnos` 含当前 transno（`*` 放行）→ timestamp ±5 分钟。
3. `DataApiController.query`：剥离 `_export*` 参数（防注入导出分页）→ `XmlConfigCacheManager.get(transno)`（命中即返，未命中从 `interface_info` 表加载 XML 并解析为 `InterfaceConfig` 缓存）。
4. `XmlEngine.executeWithConfig` 四阶段：
   - **参数校验**：`requestData.params` 必填项检查（带 defaultValue 兜底）。
   - **注册内联数据源**：XML 中 `<datasource>` 经 `DataSourceRegistrar` 回调动态注册。
   - **查询编排（DAG）**：`QueryOrchestrator.orchestrate` 按 `depends` 拓扑用 `CompletableFuture` 并行执行；SQL 类经 `DynamicSqlHandler`（SpEL）→ `PaginationHandler`（改写）→ `SqlExecutor`（push 数据源 → JdbcTemplate）；后继查询可通过 `previousResults` 取上游结果。
   - **结果映射**：`ResultMapper` 按 `resultMap` 重命名/函数加工，按 `responseData` 组装最终结构。
5. `GlobalResponseBodyAdvice` 统一包装为 `ApiResponse`（`code=0000`）；异常由 `GlobalExceptionHandler` 转 `ApiResponse` + 业务错误码。

**管理域数据流**：前端 `request.js` 注入 `Admin-Token` → `AdminAuthInterceptor` 解析并注入用户属性 → Controller 加 `@RequireRole` → `RoleCheckAspect` 校验 → MyBatis-Plus Service → Mapper → MySQL。

### 设计模式

| 模式 | 落地 | 位置 |
|------|------|------|
| **门面（Facade）** | `XmlEngine` 统一对外暴露四阶段执行，屏蔽内部 parser/orchestrator/executor 细节 | `XmlEngine.java` |
| **策略（Strategy）** | 按 `query.type`（mysql/doris/oracle/postgresql/http/dubbo/mongo）分派到不同 Executor；`PaginationConfig.PaginationMode`（OPTIMIZED/CURSOR）两种分页策略 | `XmlEngine.executeQuery`、`PaginationHandler` |
| **DAG 编排 / 模板方法** | `QueryOrchestrator` 以 `depends` 声明构建依赖图，`CompletableFuture.allOf` 编排；`XmlEngine.executeWithConfig` 固定四阶段模板 | `QueryOrchestrator.java` |
| **注册表（Registry）** | `FunctionRegistry` 静态注册内置函数；`XmlConfigCacheManager` 注册 `transno→config` | `FunctionRegistry.java`、`XmlConfigCacheManager.java` |
| **解耦接口（Dependency Inversion）** | `dsp-common` 定义 `DataQueryService` / `XmlConfigCacheInvalidator`，`dsp-engine` 实现，让上层服务不直接耦合 engine 内部类 | `common/service/*`、`engine/service/DataQueryServiceImpl` |
| **AOP 横切** | 鉴权/RBAC/审计全部切面化，业务代码零侵入 | `*Aspect.java` |
| **缓存 Aside** | `XmlConfigCacheManager` 读穿透 + 主动失效（admin 编辑后调 `invalidate`）+ 定时全量刷新 | `XmlConfigCacheManager.java` |
| **回调（Callback）** | `DataSourceRegistrar` 由服务侧注入实现，engine 通过回调注册内联数据源 | `engine/DataSourceRegistrar.java` + `InlineDataSourceRegistrar.java` |
| **统一报文/异常** | `ApiRequest`/`ApiResponse` + `BusinessException(ErrorCode)` + `GlobalExceptionHandler` | `common/model/*`、`common/exception/*` |

### 解耦接口

| 接口 | 定义位置 | 实现位置 | 消费者 |
|------|---------|---------|--------|
| `DataQueryService` | `dsp-common/.../common/service/DataQueryService.java` | `dsp-engine/.../engine/service/DataQueryServiceImpl.java` | `dsp-offline-service`（离线导出复用引擎，不直接 import engine 包） |
| `XmlConfigCacheInvalidator` | `dsp-common/.../common/service/XmlConfigCacheInvalidator.java` | `dsp-engine/.../engine/cache/XmlConfigCacheManager.java` | `dsp-admin-service`（接口编辑/发布后使缓存失效） |
| `DataSourceRegistrar` | `dsp-engine/.../engine/DataSourceRegistrar.java` | `dsp-data-service/.../dataservice/InlineDataSourceRegistrar.java` | `XmlEngine`（通过回调注册 XML 内联数据源，避免 engine 反向依赖 service 层） |

> 关键意义：`dsp-engine` 通过这套接口避免了对上层服务模块的循环依赖——engine 定义回调/服务接口，由上层服务注入实现。
