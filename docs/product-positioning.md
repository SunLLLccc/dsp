# DSP 产品定位

## 一句话定位

DSP 是面向企业查询型数据接口的 AI 辅助设计、受控生成与全生命周期治理平台。

平台把自然语言数据需求转化为有表结构依据、受模板约束、可验证、可审批、可发布和可审计的数据接口。AI 负责理解与生成候选内容，平台负责约束、确认、版本、发布、授权、审计和运行治理。

## 用户与痛点

### 目标用户

- 数据接口建设者：负责把业务查询需求落成稳定接口的开发、数据平台或集成团队。
- 接口审核者：需要确认接口字段、SQL、数据源、版本和发布风险的负责人。
- 接口使用方：通过授权 Token 调用查询接口的业务系统或应用。
- 平台运维者：关注发布状态、调用审计、缓存刷新、数据源配置和运行稳定性。

### 主要痛点

- 业务需求常以自然语言或文档表达，接口定义、SQL、字段映射和响应结构之间容易丢失上下文。
- 普通 Text2SQL 容易在缺少表结构依据时生成看似合理但不可上线的 SQL。
- 直接让 AI 生成配置缺少边界，XML 结构、SQL 安全和模板一致性难以治理。
- 查询接口上线不是一次生成动作，还需要确认、版本、审批、发布、授权、审计和运行缓存管理。
- 多数据源、多段查询、分页、动态条件和结果映射需要统一执行底座，而不是每个接口重复写控制层代码。

## 核心价值主张

DSP 的价值在于把 AI 生成纳入企业接口治理流程，而不是用 AI 或 XML 替代工程判断。

- **受依据约束**：Text2SQL 必须有 Schema Evidence，至少包含表名和字段信息；没有依据时系统追问而不是生成 SQL。
- **受模板约束**：XML/JSON 配置来自模板选择与填充，AI 不自由编写任意 XML。
- **人在回路**：需求、接口定义、SQL、模板、XML/JSON 和发布前均可确认、修改或回退。
- **可验证**：生成 XML 会经过模板路径校验、SQL 安全校验和导入结构校验。
- **可治理**：接口进入版本、审批、发布、授权、审计、缓存失效和运行查询链路。
- **可复用执行底座**：XML DSL、动态 SQL、分页、DAG 查询编排和多数据源执行器承担运行时执行。

## 产品边界

### 当前聚焦

- 查询型、数据聚合型接口。
- SQL 类接口的 Text2API 生成链路。
- 项目知识问答与本地文档/源码检索辅助。
- 接口版本、审批、导入发布、授权、审计和缓存治理。
- XML DSL 驱动的查询编排与结果映射执行。

### 明确不覆盖

- 复杂交易、写操作、长事务或强一致业务流程。
- 通用后端代码生成、Controller/Service/Mapper 全栈代码生成。
- 任意数据库变更、DDL、DML 或绕过安全校验的 SQL 生成。
- AI 自动替代审批、自动确认字段含义或自动保证业务正确性。
- 无表结构依据时直接生成可上线 SQL。

## 核心工作流

```mermaid
flowchart LR
    A["业务需求"] --> B["接口定义"]
    B --> C["Schema Evidence"]
    C --> D["Text2SQL"]
    D --> E["模板选择"]
    E --> F["XML/JSON 配置生成"]
    F --> G["安全校验与复核"]
    G --> H["审批或导入发布"]
    H --> I["授权、审计与运行治理"]
```

### 已实现链路

- Text2API 草稿：保存需求、接口定义、Schema Evidence、SQL 草稿、模板选择、XML/JSON 草稿和发布错误。
- 阶段生成：接口定义、Text2SQL、模板选择、XML/JSON 生成均有阶段前置条件。
- Schema Evidence 门禁：无有效表名和字段依据时不调用 AI 生成 SQL。
- 模板填充：模板路径限制在 `template/` 目录，SQL 段数与模板 query 节点数必须精确匹配。
- 发布：Text2API 通过 `ConfigImportService.importConfig` 导入并直接发布，失败记录 `publishError` 并允许重试。
- 传统接口治理：接口版本支持保存、提交审批、通过发布、驳回、撤回和下线。
- 运行治理：已发布接口通过缓存加载执行，发布/下线触发缓存失效；管理端和数据服务记录审计日志。

## 与相近系统的区别

### 与普通 Text2SQL 的区别

普通 Text2SQL 关注“从自然语言到 SQL”。DSP 关注“从数据需求到可治理接口”。SQL 只是链路中的一个阶段，必须有 Schema Evidence，并且后续还要经过模板选择、XML/JSON 生成、校验、确认、发布和审计。

### 与代码生成器的区别

代码生成器通常输出源代码，后续由开发者接管编译、部署和维护。DSP 不以生成后端代码为目标，而是把接口定义、SQL、XML 配置和发布状态保存在平台中，由统一运行时解释执行，并纳入版本、授权和审计。

### 与传统低代码平台的区别

传统低代码平台常强调少写代码或拖拽配置。DSP 的重点不是“少写 XML”，而是让 AI 生成结果被依据、模板、人审和平台治理约束。XML DSL 与 DAG 编排是执行底座，服务于查询接口的受控生成与运行治理。

## 当前能力与未来规划

### 当前已实现

- AI 助手：AgentScope Java 2.0 RC5 接入，SSE 流式问答，本地文档优先检索，文档不足时源码兜底。
- Text2API：需求、接口定义、Schema Evidence、Text2SQL、模板选择、XML/JSON 生成、确认/回退、导入发布。
- 元数据辅助：后端具备读取已配置数据源表和字段元数据的服务，可作为后续入口对接的 Schema Evidence 来源。
- 模板约束：读取 `ai-assets/template-index.json` 选择 SQL 类模板，并填充 `template/` 下 XML 模板。
- 安全校验：SQL 只读关键字、分号、注释、orderBy 字段和 XML 解析校验。
- 治理能力：接口版本、审批发布、应用授权、审计日志、运行缓存、在线/离线导出。

### 未来规划

- 更细粒度的 Schema Evidence 管理：字段含义、关联关系、示例值和数据权限说明。
- 向量检索/RAG：在本地文档检索基础上增强召回，但仍保持来源引用和人工确认。
- 模板体系扩展：补充更多 SQL 查询模板，例如 2-query 分页模板、多结果集模板和常见聚合模板。
- 更柔性的 AI 追问策略：必填字段已满足时允许落库，同时把未确认问题作为复核提示展示。
- WebSocket 或更强的协作协议：在需要双向协作、心跳和多路复用时评估替换 SSE。
- 发布前自动化验证增强：生成后自动构造示例请求、执行只读调试并沉淀校验报告。

## 可衡量的产品指标

- 需求到可复核接口草稿的平均耗时。
- Text2SQL 因缺少 Schema Evidence 被拦截的次数与比例。
- AI 生成结果在接口定义、SQL、模板、XML/JSON 阶段的人工修改次数。
- 模板匹配成功率、模板缺口类型和失败原因分布。
- 发布成功率、发布失败重试次数、`publishError` 类型分布。
- 审批通过率、驳回原因和从提交到发布的平均周期。
- 已发布接口调用量、错误率、平均响应时间和慢查询数量。
- 授权应用数、接口授权覆盖率和审计查询频次。
